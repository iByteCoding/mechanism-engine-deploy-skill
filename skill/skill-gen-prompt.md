# 机制能力部署 Skill 生成提示词

## 适用场景

当某个业务模块中已集成了某种机制能力（如表单引擎、附件机制、审批流程等），需要将其提炼为可复用 Skill 时使用，以方便在其他模块中便捷集成该机制。

---

## 标准提示词模板

```
在 @{案例模块}/ 中已部署 @{机制模块}/，需要完成以下任务（需要的权限直接获取）：

1. 从 @{案例模块} 中全面提取所有使用该机制的相关内容，包括但不限于配置文件、实现代码、调用逻辑及相关依赖。

2. {案例模块} 仅作为参考案例，提取的代码和内容中不得出现案例模块相关的业务代码，所有示例代码使用通用占位符（如 {Module}、{module}、{模块Key} 等）替代。

3. 基于提取的内容，开发一个可复用的"{机制名称}部署" Skill，以便在其他模块中便捷集成该机制。

4. 生成完整的 SKILL.md，结构如下（按序包含以下各节）：
   - **frontmatter**：只含 name 和 description 两个字段，name 与上层目录名一致（如：MK表单引擎部署，不需要引号包裹）
   - **一、实体概览**（如有多种实体类型）：表格列出各实体类型的识别特征、典型场景、核心接口/基类、关联字段及示例类名；表格下方以"**关键约束**"列出各类型最容易踩坑的注意事项
   - **二、部署流程**：按实际机制拆分若干步骤，通常包含 Maven 依赖（直接在 SKILL.md 中展开，含完整 xml 片段）和执行部署（表格展示各机制/子步骤的名称、参考文档链接、依赖关系、步骤数说明）；如机制部署前需要收集模块信息或用户确认，可按需加入对应步骤
   - **四、部署验证清单**：以 `- [ ]` 格式逐项列出可核查的验证点
   - **五、代码注释规范**：表格列出各代码位置（接口方法、Service 方法、Entity 字段、Controller 端点、类级注释等）及对应注释要求；表格后补充"行内注释原则"说明
   - **六、占位符说明**：表格列出所有占位符、含义及示例值

5. 将详细内容整理到 references 目录，遵循以下要求：
   - **首先生成 ref-00-bundle-spec.md**：汇总该 Skill 所有 ref 文件涉及的跨模块包导入，按依赖层分组（如：JDK/标准库、Lombok、Hutool、Spring Framework、Swagger、框架基础层、框架业务层等），每组用三级标题区分；作为全局约束文件，所有其他 ref 文件均须遵守，ref 内代码块只保留本机制新建类的自模块导入
   - **其余 ref 文件按以下两种方式拆分**：
     - **有 Entity 定义或调整时**：以 Entity 为切入点组织一个 ref 文件，在该文件中依次补全与该 Entity 相关的 VO、API、Repository、Service、Controller 等各层代码；文件后缀：机制能力用 `-spec`（如 `ref-01-category-spec.md`），引擎集成用 `-integration`（如 `ref-01-form-integration.md`）
     - **无 Entity 时**：将该能力单独定义为一个 ref 文件，描述其独立能力（如工具类、枚举、配置、i18n 等）；后缀同上规则（如 `ref-03-util-spec.md`）
   - 每个 ref 文件内部，每个代码单元使用"**核心要点**"小节（含要点说明和表格）+ 代码块 + 末尾"**注意事项**"列表的结构
   - 代码示例需包含完整注释，说明每个注解参数、关键逻辑的作用和原因；行内注释说明"为什么"而非"做什么"
   - 确保代码和配置具有通用性和可移植性

文件和目录结构规范：
{机制名称}部署/
  ├── SKILL.md
  └── references/
      ├── ref-00-bundle-spec.md            ← 全局包导入规范（所有 ref 均须遵守）
      ├── ref-01-{Entity语义}-spec.md          ← 机制能力型（Entity + 全栈）
      ├── ref-02-{Entity语义}-integration.md  ← 引擎集成型（Entity + 全栈）
      └── ref-0N-{能力语义}-spec.md            ← 无 Entity 的独立能力（工具类/枚举/配置等）
```
